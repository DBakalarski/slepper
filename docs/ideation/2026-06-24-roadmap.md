# Roadmap rozbudowy Sleeper (post-MVP)

> Odznaczalny plan wywiedziony z ideacji (`2026-06-24-open-ideation.md`). Fazy ułożone wg
> zależności i stosunku wartość/koszt. **Każdy nietrywialny ficzer i tak przejdź przez
> `/dev-brainstorm` → `/dev-plan` przed kodem** — checkboxy tutaj to backlog/postęp, nie
> zastępują planu technicznego (Implementation Units).
>
> **Definition of Done (każde zadanie kodowe):** `pnpm web:build:check` zielony
> (tsc + lint + test + invariants + build) + nowy kod ma test + commit + log w `docs/commits/`.
> Czas/daty zawsze przez `lib/time.ts` (TZ-safe), nigdy surowy `Date`/`setHours`.

Legenda: `[ ]` do zrobienia · `[~]` w toku · `[x]` zrobione · 🔴 wysoka złożoność · 🟡 średnia · 🟢 niska

## Faza 0 — Szybkie wygrane i fundamenty 🟢

Niezależne, niskiego ryzyka, odblokowują późniejsze fazy.

- [ ] **Quick-undo po "Zakończ sen"** — toast "Przywróć" (3 s) cofający `end_at = null` zamiast dialogu potwierdzenia
- [ ] **Helper agregacji snu** — wydzielić jeden moduł (`lib/sleep-aggregation.ts`?) liczący: suma snu/dobę, liczba+długość drzemek, σ pory zasypiania; oparty wyłącznie o `lib/time.ts`
  - [ ] test jednostkowy na fixtures (dzień normalny, sesja cross-midnight, dzień bez snu)
- [ ] **Decyzja: biblioteka wykresów** — wybrać kompatybilną z react-native-web, sprawdzić bundlephobia, zaplanować lazy-load ekranu (zależność dla Fazy 1)

## Faza 1 — Dashboard snu i trendy (Pomysł #1, RECOMMENDED) 🟡

Zastępuje placeholder `stats.tsx`. Największy stosunek wartości do kosztu.

- [ ] `/dev-brainstorm` — doprecyzować zakres MVP dashboardu (jakie wykresy, jaki okres domyślny)
- [ ] Widok tygodniowy: suma snu/dobę (wykres słupkowy 7 dni)
- [ ] Widok miesięczny / wybór zakresu
- [ ] Wskaźnik regularności pór zasypiania (σ bedtime)
- [ ] Porównanie z normą dla wieku (re-użyć `children.birth_date` + dane norm z silnika)
- [ ] "Sleep score / forma snu" z ostatnich 2–3 dni
- [ ] Lazy-load ekranu (code splitting, by nie obciążać Home)
- [ ] Stany empty / loading / error
- [ ] Testy + walidacja + commit

## Faza 2 — Tagi kontekstu snu + korelacje (Pomysł #4, WORTH_EXPLORING) 🟢

Zasila dashboard realnym wglądem. Robić po/równolegle z Fazą 1.

- [ ] Migracja: kolumna `sessions.tags` (JSON array) + ew. tabela słownika tagów
- [ ] Predefiniowana lista tagów (ząbkowanie, choroba, nowa lokalizacja, hałas, zmiana opieki, …)
- [ ] Selektor tagów w modalu edycji sesji — w pełni opcjonalny, jednodotykowy (nie zamienić w formularz)
- [ ] Dashboard: filtr/porównanie średniej długości snu z tagiem vs bez
- [ ] Testy + walidacja + commit

## Faza 3 — Eksport / raport dla pediatry (Pomysł #6, WORTH_EXPLORING) 🟡

Re-używa agregacji z Fazy 1. Zacząć od samego CSV.

- [ ] Eksport CSV (data, długość, typ, notatki, tagi) — pobranie z poziomu History/Dashboard
- [ ] (Opcjonalnie, osobna iteracja) Raport agregowany 2-tyg/miesiąc (PDF lub udostępnialny widok) przez Edge Function
- [ ] Testy + walidacja + commit
- _Poza scope: integracja Apple Health (native-only) i Google Sheets sync (nad-inżynieria) — odrzucone w ideacji._

## Faza 4 — Współpraca w czasie rzeczywistym (Pomysł #3, RECOMMENDED) 🟡

Event realtime już płynie; brakuje warstwy UI + obsługi konfliktu.

- [ ] Banner/toast "Sesja zmieniona przez partnera" (domknąć TODO w `useRealtimeSessions.ts`)
- [ ] Wskaźnik obecności partnera (online / offline od X) — Supabase Realtime presence
- [ ] Detekcja konfliktu edycji tej samej sesji na dwóch urządzeniach
  - [ ] field-level merge (preferuj remote dla zmienionych pól, zachowaj pole aktualnie edytowane przez usera)
  - [ ] mini-diff w UI
- [ ] Testy + walidacja + commit

## Faza 5 — Persystencja offline-first (Pomysł #2, RECOMMENDED) 🔴

Najcięższe technicznie. **Najpierw zmierz realny odsetek użyć offline** (Faza 6 monitoring) zanim zbudujesz pełną kolejkę.

- [ ] `/dev-brainstorm` + `/dev-plan` (zmiana warstwy persystencji — wymaga planu IU)
- [ ] Idempotency key dla mutacji START/STOP/EDIT (ochrona przed duplikatem przy retry)
- [ ] Kolejka mutacji offline w IndexedDB + replay po powrocie sieci
- [ ] Conflict resolution przy replay (min. last-write-wins)
- [ ] Cache historii w IndexedDB jako fallback offline (zamiast pustego spinnera)
- [ ] Stan połączenia realtime + reconnect z exponential backoff + badge w UI
- [ ] Testy (w tym scenariusze offline→online, podwójny tap) + walidacja + commit

## Faza 6 — Proaktywne alerty (Pomysł #5, WORTH_EXPLORING) 🔴

Zależy od Fazy 1 (heurystyki) i od infrastruktury powiadomień.

- [ ] **Spike:** zweryfikować wsparcie web push na PWA (zwł. iOS Safari) — decyzja in-app vs push
- [ ] Heurystyki: regresja snu (−20% tydzień do tygodnia), niespójne pory (σ > 60 min), okno "przemęczenia"
- [ ] Wyzwalanie: Edge Function (cron) lub logika in-app
- [ ] Alert in-app z konkretną akcją; progi i ton komunikatów ostrożne (nie obwiniać rodzica)
- [ ] Testy heurystyk na fixtures + walidacja + commit

## Faza 7 — Tryb nocny / obsługa jedną ręką (Pomysł #7, WORTH_EXPLORING) 🟡

Klaster UX — ciąć na małe, niezależne usprawnienia.

- [ ] Bardzo ciemny tryb nocny (AMOLED-friendly), oddzielny od system dark mode
- [ ] Przyciemnianie Sleep Fullscreen w czasie trwania sesji
- [ ] Hold-tap (~1 s) na krytycznych akcjach (anty-przypadkowe kliknięcie)
- [ ] Wolniejszy tick timera w tle (oszczędność baterii przy telefonie leżącym 6–8 h)
- [ ] Testy/manual checklist + walidacja + commit
- _Poza scope: haptyka (web — Vibration API zawodny/niedostępny). Quick-undo już w Fazie 0._
