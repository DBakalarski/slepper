## Powod / kontekst

Cykl 2 fix po Code Review (5 agentow). Wykryto NOWY P1 (race
condition w atomic delete osieroconej rodziny). User wybral "fix
wszystko" (P1 + 16 P2). Plus 1 z 7 P2 cyklu 1 niepokrytych
(refetch po SIGNED_IN) zostal swiadomie pominiety — cykl 2 powiedzial
ze fix przez fallback wystarczy, a manual refetch by user-action
jest acceptable.

Decyzje projektowe:
- Atomic delete via `NOT EXISTS` w jednym DELETE statement zamiast
  IF/THEN check — eliminuje TOCTOU. Postgres gwarantuje atomicity
  pojedynczego statementu w Read Committed.
- Last-owner guard w `accept_invitation` (nie w DELETE policy) — RPC
  to centralny punkt do leave-family flow w MVP. Po Fazie 2+ mozna
  dodac analogiczny guard do DELETE policy gdy bedzie UI "Opusc
  rodzine".
- 1-query embed z relational select — Supabase obsluguje `family:
  families!inner(...)` z zagniezdzonymi members:family_members.
  Inner join wymusza poprawne FK matching.
- focusManager.setFocused(state === 'active') — standardowy pattern
  TanStack Query + React Native, dziala dla wszystkich aktualnych
  i przyszlych queries (sessions w Fazie 2+, realtime w Fazie 4).

Swiadomie pominiete:
- Trigger fires bez `email_confirmed_at` (P2 security) — email
  confirm OFF w decyzji MVP, sprawdzanie nie ma sensu. Zostawione.
- AuthContextValue narrowing nie dociera (P2 arch debate) — consumers
  uzywaja optional chain, acceptable dla MVP. Dyscriminated union
  zaplaca sie w (app)/_layout i (auth)/_layout.
- Const literals dla LOADING/SIGNED_OUT (P3 micro-opt) — zero ROI.
