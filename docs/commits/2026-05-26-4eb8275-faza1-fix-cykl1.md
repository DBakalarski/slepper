- **Migracja 0005_consent_flow.sql:**
  - `handle_new_user()` trigger ZMIANA: nie acceptuje invitations
    auto-magicznie. Tylko tworzy nowa rodzine dla nowego usera.
    Likwiduje pre-claim invitation exploit.
  - Nowy RPC `get_my_pending_invitations()` SECURITY DEFINER — zwraca

    rodziny.

    explicit consent. Sprawdza ze invitation email == auth.email().

  - Nowy RPC `ensure_family()` SECURITY DEFINER idempotent — fallback
    dla user bez membership.

## Powod / kontekst

Po Code Review Fazy 1 (5 agentow rownolegle przez /dev-docs-review)
ujawniono 3 P1: pre-claim exploit, partner-already-exists deadlock,
ensure_family ślepy zaułek. User wybral fix-now strategy (cykl 1/2).

Decyzje projektowe:
- Consent flow zamiast auto-accept — fundamentalna zmiana modelu
  trust. Trigger zostawia user w solo-rodzinie do explicit consent
  via RPC. Eliminuje calosc P1.1 (pre-claim) i P1.2 (partner-already-
  exists rozwiazany przez post-sign-in RPC dla istniejacych userow).
- `accept_invitation` usuwa stara rodzine jesli osierocona — chroni
  przed wyciekiem orphan families w bazie. User-id UNIQUE constraint
  wymusza atomowe przepiecie.
- Banner w index.tsx zamiast osobnego ekranu/modala — najprostsza
  UX, MVP-acceptable.

P2 pozostale (3-query waterfall PostgREST embed, cancel guard w
sign-in/sign-up, profile.tsx ekstrakcja komponentow) + P3 batch
(EMAIL_REGEX/translateAuthError extraction, password min length,
SecureStore, etc.) — do cyklu 2 lub backlog Fazy 2.
